# 채팅 서비스 PRD

## 개요

본 서비스는 사용자 간 실시간 채팅 기능을 제공한다.
채팅 방은 `DM`, `GROUP`, `PUBLIC` 타입으로 구성되며, 사용자는 채팅 방 생성, 참여, 초대, 메시지 송수신 기능을 사용할 수 있어야 한다.

## 공통 정책

- 채팅 방 타입은 `DM`, `GROUP`, `PUBLIC` 총 3가지로 구성된다.
- 모든 채팅 방은 고유한 `room_key` 를 가져야 한다.
- 모든 채팅 메시지는 최대 30일 동안 보관되어야 한다.
- 채팅 방 기본 정렬 기준은 최근 수정일 기준이어야 한다.
- 사용자는 동시에 여러 채팅 방에 연결할 수 있어야 한다.
- 메시지는 채팅 방 참여자에게 실시간으로 전달되어야 한다.

## 채팅 방 조회 기능

### 내가 참여한 채팅 방 조회

- 사용자는 자신이 참여 중인 채팅 방 목록을 조회할 수 있어야 한다.
- 사용자는 그룹 기준으로 채팅 방을 조회할 수 있어야 한다.
- 사용자가 참여 중인 모든 채팅 방은 기본적으로 `전체 그룹`에 포함되어야 한다.
- 채팅 방 목록은 최근 수정일 기준으로 정렬되어야 한다.
- 채팅 방 목록은 페이지네이션 없이 전체를 반환해야 한다.

### PUBLIC 채팅 방 조회

- 참여하지 않은 `PUBLIC` 채팅 방 목록을 별도로 조회할 수 있어야 한다.
- 모든 사용자는 `PUBLIC` 채팅 방 목록을 조회할 수 있어야 한다.
- `PUBLIC` 채팅 방은 제목 기준 검색이 가능해야 한다.
- 제목 검색은 부분 일치를 지원해야 한다.
- 검색 결과는 페이징 처리 가능해야 한다.

## 채팅 방 생성 기능

### DM 채팅 방

- `DM` 채팅 방 생성 시 상대방의 계정 아이디를 지정해야 한다.
- 계정 아이디는 외부 인증 서버에서 발급된 JWT의 식별값(`sub`)을 기준으로 한다.
- 존재하지 않는 계정 아이디로 DM 생성 시 오류를 반환해야 한다.
- 동일한 두 사용자 간 `DM` 채팅 방은 중복 생성될 수 없어야 한다.
- `DM` 채팅 방 최대 참여 인원은 2명이어야 한다.
- `DM` 채팅 방은 외부 초대 기능을 제공하지 않는다.
- `DM` 채팅 방은 다음 형식의 `room_key` 를 가져야 한다.
    - `dm:{HASH}`
    - HASH 생성 시 두 사용자의 UUID를 사전식 문자열 비교 기준으로 정렬해야 한다.
    - 정렬된 UUID를 다음 형식으로 조합해야 한다.
        - `{UUID}:{UUID}`
    - 최종 조합 문자열을 해시하여 `room_key`를 생성해야 한다.

### GROUP 채팅 방

- `GROUP` 채팅 방 생성 시 초기 참여자를 지정하지 않아야 한다.
- 채팅 방 생성자는 자동으로 참여자가 되어야 한다.
- `GROUP` 채팅 방은 초대를 통해 참여할 수 있어야 한다.
- `GROUP` 채팅 방 최대 참여 가능 인원은 10명이어야 한다.
- `GROUP` 채팅 방은 다음 형식의 `room_key` 를 가져야 한다.
    - `group:{UUID}`

### PUBLIC 채팅 방

- `PUBLIC` 채팅 방 생성 시 초기 참여자를 지정하지 않아야 한다.
- 채팅 방 생성자는 자동으로 참여자가 되어야 한다.
- `PUBLIC` 채팅 방은 누구나 참여할 수 있어야 한다.
- `PUBLIC` 채팅 방 최대 참여 가능 인원은 10명이어야 한다.
- `PUBLIC` 채팅 방은 비공개 설정이 가능해야 한다.
- 비공개 `PUBLIC` 채팅 방은 비밀번호 입력 후 참여 가능해야 한다.
- `PUBLIC` 채팅 방은 다음 형식의 `room_key` 를 가져야 한다.
    - `public:{UUID}`

## 채팅 방 참여 정책

- 이미 참여 중인 사용자는 동일한 채팅 방에 중복 참여할 수 없어야 한다.
- 채팅 방 최대 인원 초과 시 참여할 수 없어야 한다.
- 사용자는 채팅 방에서 나갈 수 있어야 한다.
- 채팅 방에서 나간 사용자는 재참여 가능해야 한다.
- `GROUP`, `PUBLIC` 채팅 방은 회원 초대 기능을 제공해야 한다.
- 초대받은 사용자는 초대를 수락 또는 거절할 수 있어야 한다.
- 이미 참여 중인 사용자는 초대할 수 없어야 한다.
- 강퇴된 사용자는 해당 채팅 방에 재참여할 수 없어야 한다.
- 강퇴된 사용자는 초대할 수 없어야 한다.

## 채팅 방 권한 정책

- `GROUP`, `PUBLIC` 채팅 방 생성자는 방장 권한을 가져야 한다.
- 방장은 채팅 방 정보를 수정할 수 있어야 한다.
- 방장은 참여자를 초대할 수 있어야 한다.
- 방장은 참여자를 강퇴할 수 있어야 한다.
- 방장이 채팅 방을 나갈 경우 다른 참여자에게 방장 권한이 자동으로 위임되어야 한다.
- 방장 위임 순서는 참여 순서 기준으로 다음 참여자에게 위임된다.
- 방장이 마지막 참여자인 경우 나가도 채팅 방은 삭제되지 않고 빈 방으로 남는다.

## 채팅 방 삭제 정책

### DM 채팅 방

- DM 채팅 방에서 나가기는 삭제가 아닌 숨김 처리로 동작해야 한다.
- 숨김 처리된 DM 채팅 방은 내 목록에서 보이지 않아야 한다.
- 상대방이 새 메시지를 전송하면 숨김이 해제되고 목록에 다시 나타나야 한다.
- 이전 메시지 기록은 유지되어야 한다.

### GROUP / PUBLIC 채팅 방

- 방장은 채팅 방을 삭제할 수 있어야 한다.
- 삭제된 채팅 방은 더 이상 조회 및 참여할 수 없어야 한다.
- 마지막 참여자가 채팅 방을 나간 경우 자동 삭제 여부는 추후 정책으로 결정 가능해야 한다.

## 그룹 기능

### 그룹 생성 및 관리

- 사용자는 채팅 방 그룹을 생성할 수 있어야 한다.
- 사용자는 최대 10개의 그룹을 생성할 수 있어야 한다.
- 그룹 최대 생성 가능 개수는 정책 변경 가능성이 존재한다.
- 사용자가 생성한 그룹 이름은 사용자 기준으로 중복될 수 없어야 한다.
- 사용자는 자신이 생성한 그룹을 수정 및 삭제할 수 있어야 한다.

### 기본 그룹 정책

- 모든 사용자는 기본적으로 `전체 그룹`을 가지고 있어야 한다.
- `전체 그룹`은 수정 및 삭제가 불가능해야 한다.

## 회원 초대 기능

- `GROUP`, `PUBLIC` 채팅 방은 회원 초대 기능을 제공해야 한다.
- 초대받은 사용자는 채팅 방 참여 가능 여부를 선택할 수 있어야 한다.

## 실시간 연결 정책

- 사용자는 동시에 여러 채팅 방에 연결할 수 있어야 한다.
- WebSocket 연결 시 JWT 인증을 통과해야 한다.
- 인증 실패 시 연결이 거부되어야 한다.
- 메시지는 채팅 방 참여자 전원에게 실시간으로 브로드캐스트되어야 한다.
- 사용자의 온라인 상태는 heartbeat 기반으로 유지되어야 한다.
- heartbeat가 일정 시간(60초) 내에 갱신되지 않으면 오프라인으로 처리되어야 한다.
- 연결 정상 종료 시 즉시 오프라인 상태로 처리되어야 한다.
- 연결 비정상 종료 시 TTL 만료 후 오프라인으로 처리되어야 한다.
- 사용자의 온라인 상태를 조회할 수 있어야 한다.
- 초대 알림 등 개인 알림은 개인 채널을 통해 실시간으로 수신할 수 있어야 한다.

## 프로필 기능

- 최초 로그인 시 기본 프로필이 자동으로 생성되어야 한다.
- 사용자는 여러 개의 프로필을 생성할 수 있어야 한다.
- 각 프로필은 닉네임을 가진다.
- 프로필 닉네임은 수정 가능해야 한다.
- 프로필은 삭제 가능해야 한다.
- 채팅 방에서 사용 중인 프로필은 삭제할 수 없어야 한다.

## 채팅 방 프로필 선택

- 사용자는 채팅 방에 입장할 때 자신의 프로필 목록 중 하나를 선택해야 한다.
- 채팅 방마다 다른 프로필을 사용할 수 있다.
- 입장 후 사용 중인 프로필은 변경 가능해야 한다.

## 메시지 기능

### 메시지 저장

- 채팅 방에 입력된 메시지는 저장되어야 한다.
- 메시지는 채팅 방 단위로 조회 가능해야 한다.
- 메시지는 기본적으로 텍스트 타입을 지원해야 한다.
- 텍스트 메시지는 최대 1,000자까지 입력 가능해야 한다.
- 이미지 및 파일 메시지 확장이 가능해야 한다.
- 본인이 보낸 메시지는 수정 가능해야 한다.
- 수정된 메시지는 수정 여부를 표시해야 한다.

### 읽음 처리

- 사용자는 채팅 방별 읽지 않은 메시지 개수를 확인할 수 있어야 한다.
- 사용자가 채팅 방에 진입하면 마지막 읽은 메시지 위치를 갱신해야 한다.
- 읽지 않은 메시지 개수는 마지막 읽은 메시지 기준으로 계산해야 한다.

### 메시지 보관 정책

- 메시지는 생성일 기준 최대 30일 동안 보존되어야 한다.
- 30일이 지난 메시지는 배치 작업을 통해 자동 삭제되어야 한다.

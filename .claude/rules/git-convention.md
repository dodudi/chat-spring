## 1. 브랜치 전략

### 브랜치 구조

```
main         # 배포 브랜치 (직접 push 금지)
develop      # 개발 통합 브랜치
feature/     # 기능 개발
fix/         # 버그 수정
hotfix/      # 운영 긴급 수정
refactor/    # 리팩토링
chore/       # 빌드, 설정 변경
```

### 브랜치 네이밍

```
{타입}/{이슈번호}-{간단한-설명}

예시)
feature/123-user-login
fix/456-order-null-pointer
hotfix/789-payment-timeout
refactor/101-user-service-cleanup
```

## 2. 커밋 메시지

### 형식

```
{타입}: {제목}

{본문} (선택)

{꼬리말} (선택)
```

### 타입

| 타입 | 설명 |
|---|---|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 리팩토링 (기능 변경 없음) |
| `test` | 테스트 코드 추가/수정 |
| `docs` | 문서 수정 |
| `style` | 포맷팅, 세미콜론 등 코드 변경 없음 |
| `chore` | 빌드, 패키지, 설정 변경 |
| `perf` | 성능 개선 |
| `revert` | 이전 커밋 되돌리기 |

### 제목 규칙

- 50자 이내
- 마침표 금지
- 명령형으로 작성 (`추가한다` ❌ → `추가` ✅)
- 한글 또는 영어 중 팀 합의된 언어로 통일

### 예시

```
# ✅ 좋은 예
feat: 회원 로그인 API 추가
fix: 주문 조회 시 NPE 수정
refactor: UserService 메서드 분리
test: OrderService 단위 테스트 추가
chore: Gradle 의존성 버전 업데이트

# ❌ 나쁜 예
update
버그 수정함.
feat: 로그인 기능을 구현하였고 JWT 토큰 발급과 리프레시 토큰 저장 로직도 함께 추가하였습니다.
```

### 본문 / 꼬리말 (필요한 경우)

```
fix: 결제 금액 계산 오류 수정

할인 쿠폰 적용 시 음수 금액이 발생하는 문제를 수정.
최소 결제 금액(0원) 이하로 내려가지 않도록 검증 로직 추가.

Fixes #456
```

꼬리말에 이슈 번호를 연결하면 GitHub에서 PR 병합 시 이슈가 자동으로 닫힙니다.

| 키워드 | 동작 |
|---|---|
| `Fixes #123` | 병합 시 이슈 자동 close |
| `Refs #123` | 이슈 참조만 (close 안 함) |
| `Related #123` | 관련 이슈 언급 |

## 3. Pull Request

### PR 제목

커밋 메시지 제목 규칙과 동일하게 작성합니다.

```
feat: 회원 로그인 API 추가
fix: 주문 조회 시 NPE 수정
```

### PR 크기

- 리뷰어가 한 번에 검토할 수 있는 크기로 유지
- **변경 파일 10개 이하**, **diff 400줄 이하** 권장
- 기능이 크다면 단계별로 PR을 나눈다

### PR 템플릿

```markdown
## 개요
<!-- 이 PR에서 무엇을 왜 변경했는지 간략히 설명 -->

## 변경 사항
- [ ] 변경 내용 1
- [ ] 변경 내용 2

## 테스트
<!-- 어떻게 테스트했는지 작성 (단위 테스트, 수동 테스트 등) -->
- [ ] 단위 테스트 추가/수정
- [ ] 로컬 환경에서 동작 확인

## 스크린샷 (선택)
<!-- API 변경이나 UI 변경이 있을 경우 첨부 -->

## 참고 사항
<!-- 리뷰어가 알아야 할 사항, 관련 이슈/문서 링크 -->
Closes #이슈번호
```

### PR 체크리스트 (작성자)

PR을 올리기 전 스스로 확인합니다.

- [ ] 브랜치가 최신 `develop` 을 기반으로 한다
- [ ] 불필요한 파일(`.idea`, `*.class` 등)이 포함되지 않았다
- [ ] 로컬에서 빌드 및 테스트가 통과한다
- [ ] 관련 테스트 코드를 작성했다
- [ ] `System.out.println`, 디버그용 코드를 제거했다
- [ ] PR 설명을 충분히 작성했다

## 4. 코드 리뷰

### 리뷰어 규칙

- PR 등록 후 **1 영업일 이내** 리뷰 시작
- 단순 스타일 지적보다 **로직, 설계, 누락된 예외 처리** 위주로 검토
- 승인(Approve)은 최소 **1명** 이상

### 리뷰 코멘트 레벨

리뷰 코멘트 앞에 레벨을 표시해 의도를 명확히 합니다.

| 레벨 | 의미 | 반영 여부 |
|---|---|---|
| `[P1]` | 반드시 수정 필요 (버그, 보안, 정책 위반) | 필수 |
| `[P2]` | 수정 권장 (가독성, 성능, 설계) | 권장 |
| `[P3]` | 선택 사항 (개인 의견, 사소한 제안) | 선택 |
| `[Q]` | 질문 (이해를 위한 확인) | 답변 필요 |
| `[NIT]` | 사소한 지적 (오타, 네이밍 등) | 선택 |

```
# 예시
[P1] 여기서 NPE가 발생할 수 있습니다. Optional로 감싸주세요.
[P2] 이 로직은 별도 메서드로 분리하면 가독성이 좋아질 것 같습니다.
[P3] 변수명을 `list` 보다 `userList`로 하면 더 명확할 것 같아요.
[Q] 이 조건이 false일 때는 어떻게 처리되나요?
[NIT] 오타: `processign` → `processing`
```

### 작성자 응답 규칙

- 모든 코멘트에 반응(답변 또는 반영 후 resolve)
- 반영하지 않는 경우 이유를 설명하고 리뷰어와 합의
- 수정 후 `[반영완료]` 또는 커밋 링크로 답변

# Git 규칙

이 파일은 브랜치, 커밋, PR 작성 시 항상 따라야 할 규칙을 정의한다.

---

## 브랜치 전략

| 브랜치 | 용도 |
|--------|------|
| `main` | 배포 가능한 상태 유지 |
| `feat/{기능명}` | 새 기능 개발 |
| `fix/{버그명}` | 버그 수정 |
| `refactor/{대상}` | 리팩토링 |
| `chore/{작업명}` | 빌드, 설정, 의존성 등 코드 외 작업 |

```
feat/user-login
fix/token-expiry
refactor/exception-handler
chore/update-dependencies
```

`main`에 직접 push하지 않는다. 모든 변경은 브랜치 → PR → merge 순서로 진행한다.

---

## 커밋 메시지

```
{type}: {제목}

{본문 — 선택}
```

### type 목록

| type | 사용 시점 |
|------|----------|
| `feat` | 새 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 동작 변경 없는 코드 개선 |
| `test` | 테스트 추가·수정 |
| `chore` | 빌드, 설정, 의존성 |
| `docs` | 문서 수정 |

### 규칙

- 제목은 50자 이내, 명령형으로 작성 (`추가한다` X → `추가`)
- 제목 끝에 마침표 없음
- 본문이 필요한 경우 제목과 빈 줄로 구분, WHY 위주로 작성

```
// ✅ 올바른 예
feat: 사용자 로그인 API 추가
fix: 토큰 만료 시 401 대신 500 반환되는 오류 수정
chore: PostgreSQL 드라이버 의존성 추가

// ❌ 잘못된 예
update user                  // type 없음
feat: 사용자 로그인 API를 추가했습니다.  // 명령형 아님, 마침표
```

---

## PR 규칙

- 제목은 커밋 메시지 형식과 동일하게 작성
- 하나의 PR은 하나의 관심사만 다룬다
- PR 본문에 변경 내용과 테스트 방법을 간략히 기술한다
- 리뷰 없이 `main`에 merge하지 않는다

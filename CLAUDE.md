# CLAUDE.md

LLM의 흔한 코딩 실수를 줄이기 위한 행동 지침. 프로젝트별 지침과 합쳐서 사용한다.

**트레이드오프:** 이 지침은 속도보다 신중함을 우선한다. 사소한 작업은 상황에 맞게 판단한다.

## 1. 코딩 전에 생각하기

**가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것.**

구현 전에:
- 작성하려는 내용이 이미 존재하는 파일에 있는지 먼저 확인한다.
- 가정한 내용을 명시적으로 밝힌다. 불확실하면 질문한다.
- 해석이 여러 가지라면 제시한다 — 조용히 하나를 고르지 않는다.
- 더 단순한 방법이 있다면 말한다. 필요할 때는 반론을 제기한다.
- 불명확한 것이 있으면 멈춘다. 무엇이 혼란스러운지 명시하고 질문한다.

## 2. 단순함 우선

**문제를 해결하는 최소한의 코드. 추측성 코드는 작성하지 않는다.**

- 요청하지 않은 기능은 추가하지 않는다.
- 단일 사용 코드에 추상화를 도입하지 않는다.
- 요청하지 않은 "유연성"이나 "설정 가능성"을 추가하지 않는다.
- 불가능한 시나리오에 대한 에러 처리를 추가하지 않는다.
- 200줄로 작성했는데 50줄로 가능하다면 다시 작성한다.

스스로 물어본다: "시니어 엔지니어가 이걸 보고 과하다고 할까?" 그렇다면 단순화한다.

## 3. 외과적 변경

**꼭 필요한 것만 건드린다. 본인이 만든 혼란만 정리한다.**

기존 코드를 수정할 때:
- 인접한 코드, 주석, 포맷을 "개선"하지 않는다.
- 문제없는 코드를 리팩토링하지 않는다.
- 본인이라면 다르게 했더라도 기존 스타일을 유지한다.
- 관련 없는 데드 코드를 발견하면 언급한다 — 삭제하지 않는다.

본인의 변경이 고아를 만들었을 때:
- 본인의 변경으로 인해 사용되지 않게 된 import/변수/함수는 제거한다.
- 기존에 있던 데드 코드는 요청받지 않는 한 제거하지 않는다.

기준: 변경된 모든 줄이 사용자의 요청으로 직접 추적 가능해야 한다.

## 4. 목표 중심 실행

**성공 기준을 정의한다. 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 전환한다:
- "유효성 검사 추가" → "잘못된 입력에 대한 테스트를 작성하고, 통과시킨다"
- "버그 수정" → "버그를 재현하는 테스트를 작성하고, 통과시킨다"
- "X 리팩토링" → "리팩토링 전후로 테스트가 통과하는지 확인한다"

여러 단계 작업은 간략한 계획을 먼저 제시한다:
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

명확한 성공 기준이 있어야 독립적으로 반복 실행할 수 있다. 기준이 약하면 ("작동하게 만들어") 계속 확인이 필요하다.

## 6. 프로젝트 문서

신규 기능 추가·문제 해결·설정 변경 시 아래 해당 파일을 커밋 전에 반드시 갱신한다.

| 파일 | 내용 | 갱신 시점 |
|------|------|---------|
| `.claude/docs/history.md` | 기능 구현 상태 (spec 대비 현황) | 기능 추가·변경 시 |
| `.claude/docs/troubleshooting.md` | 트러블슈팅 기록 | 오류 발생 시·해결 시 |
| `.claude/docs/pr-review-setup.md` | PR 자동 코드 리뷰 설정 | 리뷰 워크플로 변경 시 |
| `.claude/specs/chat-spec.md` | 프로젝트 명세 | 요구사항 변경 시 |

### 트러블슈팅 기록 원칙

**문제가 발생하거나 수정이 이루어질 때마다 즉시 `.claude/docs/troubleshooting.md`에 기록한다.** 커밋 시점을 기다리지 않는다.

기록 대상:
- 오류 발생 → 원인 파악 전이라도 증상과 시도한 방법을 즉시 기록
- 설정 변경 → 변경 전·후 값과 이유를 기록
- 해결 완료 → 근본 원인과 해결책을 기록

기록 형식:
```markdown
## [날짜] 제목

**증상**: 어떤 오류가 발생했는가
**원인**: 왜 발생했는가
**해결**: 무엇을 어떻게 바꿨는가
```
